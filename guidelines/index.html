As Diretrizes de Acessibilidade para Conteúdo Web (WCAG) 2.2 abrangem uma vasta gama de recomendações para tornar os conteúdos Web mais acessíveis. Seguir estas diretrizes tornará os conteúdos mais acessíveis a uma gama mais ampla de pessoas com incapacidade, incluindo adaptações para cegueira e baixa visão, surdez e perda auditiva, limitações motoras, incapacidades da fala, fotossensibilidade e combinações destas condições, bem como algumas adaptações para dificuldades de aprendizagem e limitações cognitivas; contudo, não responderá a todas as necessidades dos utilizadores com estas incapacidades. Estas diretrizes abordam a acessibilidade de conteúdos Web em qualquer tipo de dispositivo (incluindo computadores de secretária, computadores portáteis, quiosques e dispositivos móveis). Seguir estas diretrizes também tornará frequentemente os conteúdos Web mais utilizáveis para os utilizadores em geral.

Os critérios de sucesso das WCAG 2.2 são redigidos como declarações testáveis que não dependem de tecnologias específicas. As orientações sobre como satisfazer os critérios de sucesso em tecnologias específicas, bem como informações gerais sobre a interpretação dos critérios de sucesso, são fornecidas em documentos separados. Consulte a Visão Geral das Diretrizes de Acessibilidade para Conteúdo Web (WCAG) para uma introdução e ligações para materiais técnicos e educativos sobre as WCAG.

As WCAG 2.2 alargam as Diretrizes de Acessibilidade para Conteúdo Web 2.1 [[WCAG21]], publicadas como Recomendação do W3C em junho de 2018. O conteúdo conforme com as WCAG 2.2 também está conforme com as WCAG 2.0 e as WCAG 2.1. O Working Group pretende que, para políticas que exijam conformidade com as WCAG 2.0 ou WCAG 2.1, as WCAG 2.2 possam constituir um meio alternativo de conformidade. A publicação das WCAG 2.2 não descontinua nem substitui as WCAG 2.0 ou as WCAG 2.1. Embora as WCAG 2.0 e WCAG 2.1 continuem a ser Recomendações do W3C, o W3C aconselha a utilização das WCAG 2.2 para maximizar a aplicabilidade futura dos esforços de acessibilidade. O W3C também incentiva a utilização da versão mais atual das WCAG no desenvolvimento ou atualização de políticas de acessibilidade Web.

Para comentar, crie uma issue no repositório GitHub das WCAG do W3C. Embora os critérios de sucesso propostos neste documento façam referência a issues que acompanham a discussão, o Working Group solicita que os comentários públicos sejam submetidos como novas issues, uma issue por cada comentário distinto. A criação de uma conta GitHub para submeter issues é gratuita. Se não for viável submeter issues no GitHub, envie um email para public-agwg-comments@w3.org (arquivo de comentários).

Introdução

Contexto das WCAG 2

As Diretrizes de Acessibilidade para Conteúdo Web (WCAG) 2.2 definem como tornar os conteúdos Web mais acessíveis para pessoas com incapacidade. A acessibilidade abrange uma vasta gama de incapacidades, incluindo incapacidades visuais, auditivas, físicas, da fala, cognitivas, linguísticas, de aprendizagem e neurológicas. Embora estas diretrizes abranjam uma ampla variedade de questões, não conseguem responder às necessidades de pessoas com todos os tipos, graus e combinações de incapacidade. Estas diretrizes também tornam os conteúdos Web mais utilizáveis por pessoas idosas com capacidades em mudança devido ao envelhecimento e frequentemente melhoram a usabilidade para os utilizadores em geral.

As WCAG 2.2 são desenvolvidas através do processo do W3C, em cooperação com indivíduos e organizações de todo o mundo, com o objetivo de fornecer uma norma comum de acessibilidade para conteúdos Web que satisfaça as necessidades de indivíduos, organizações e governos a nível internacional. As WCAG 2.2 baseiam-se nas WCAG 2.0 [[WCAG20]] e nas WCAG 2.1 [[WCAG21]], que, por sua vez, se basearam nas WCAG 1.0 [[WAI-WEBCONTENT]], e foram concebidas para serem amplamente aplicáveis a diferentes tecnologias Web, atuais e futuras, bem como para serem testáveis através de uma combinação de testes automáticos e avaliação humana. Para uma introdução às WCAG, consulte a Visão Geral das Diretrizes de Acessibilidade para Conteúdo Web (WCAG).

Foram encontrados desafios significativos na definição de critérios adicionais para responder às incapacidades cognitivas, linguísticas e de aprendizagem, incluindo um curto prazo de desenvolvimento, bem como dificuldades em alcançar consenso relativamente à testabilidade, implementabilidade e considerações internacionais das propostas. O trabalho continuará nesta área em futuras versões das WCAG. Encorajamos os autores a consultar as nossas orientações suplementares sobre melhorar a inclusão de pessoas com incapacidade, incluindo pessoas com dificuldades de aprendizagem e incapacidades cognitivas, pessoas com baixa visão e outras.

A acessibilidade Web depende não só de conteúdos acessíveis, mas também de navegadores Web acessíveis e de outros agentes de utilizador. As ferramentas de autoria desempenham igualmente um papel importante na acessibilidade Web. Para uma visão geral de como estes componentes do desenvolvimento e da interação Web funcionam em conjunto, consulte:

  • Componentes Essenciais da Acessibilidade Web
  • Visão Geral das Diretrizes de Acessibilidade para Agentes de Utilizador (UAAG)
  • Visão Geral das Diretrizes de Acessibilidade para Ferramentas de Autoria (ATAG)

Quando este documento faz referência às WCAG 2, pretende referir-se a qualquer e todas as versões das WCAG cujo número comece por 2.

Camadas de Orientação das WCAG 2

Os indivíduos e organizações que utilizam as WCAG são muito diversos e incluem designers e programadores Web, responsáveis pela definição de políticas, agentes de compras, professores e estudantes. Para responder às diferentes necessidades deste público, são disponibilizadas várias camadas de orientação, incluindo princípios gerais, diretrizes gerais, critérios de sucesso testáveis e uma vasta coleção de técnicas suficientes, técnicas consultivas e falhas comuns documentadas, acompanhadas de exemplos, ligações para recursos e código.

  • Princípios - No topo encontram-se quatro princípios que constituem a base da acessibilidade Web: percetível, operável, compreensível e robusto. Consulte também Compreender os Quatro Princípios da Acessibilidade.

  • Diretrizes - Sob os princípios encontram-se as diretrizes. As 13 diretrizes definem os objetivos fundamentais para os quais os autores devem trabalhar, de forma a tornar os conteúdos mais acessíveis a utilizadores com diferentes incapacidades. As diretrizes não são testáveis, mas fornecem o enquadramento e os objetivos gerais que ajudam os autores a compreender os critérios de sucesso e a implementar melhor as técnicas.

  • Critérios de Sucesso - Para cada diretriz, são disponibilizados critérios de sucesso testáveis que permitem utilizar as WCAG 2.2 em contextos onde sejam necessários requisitos e testes de conformidade, como especificações de design, aquisições, regulamentação e acordos contratuais. Para responder às necessidades de diferentes grupos e situações, são definidos três níveis de conformidade: A (mais baixo), AA e AAA (mais elevado). Informações adicionais sobre os níveis das WCAG podem ser encontradas em Compreender os Níveis de Conformidade.

  • Técnicas Suficientes e Consultivas - Para cada uma das diretrizes e critérios de sucesso do próprio documento WCAG 2.2, o Working Group documentou também uma grande variedade de técnicas. As técnicas são informativas e dividem-se em duas categorias: as que são suficientes para cumprir os critérios de sucesso e as que são consultivas. As técnicas consultivas vão além do que é exigido pelos critérios de sucesso individuais e permitem aos autores responder melhor às diretrizes. Algumas técnicas consultivas abordam barreiras de acessibilidade que não estão cobertas pelos critérios de sucesso testáveis. Quando são conhecidas falhas comuns, estas também são documentadas. Consulte igualmente Técnicas Suficientes e Consultivas em Compreender as WCAG 2.2.

Todas estas camadas de orientação (princípios, diretrizes, critérios de sucesso e técnicas suficientes e consultivas) funcionam em conjunto para fornecer orientações sobre como tornar os conteúdos mais acessíveis. Os autores são encorajados a consultar e aplicar todas as camadas que lhes seja possível, incluindo as técnicas consultivas, de forma a responder da melhor forma às necessidades do maior número possível de utilizadores.

Note-se que mesmo conteúdos conformes com o nível mais elevado (AAA) não serão acessíveis a indivíduos com todos os tipos, graus ou combinações de incapacidade, particularmente nas áreas cognitiva, linguística e da aprendizagem. Os autores são incentivados a considerar toda a gama de técnicas, incluindo as técnicas consultivas, Tornar o Conteúdo Utilizável para Pessoas com Incapacidades Cognitivas e Dificuldades de Aprendizagem, bem como a procurar aconselhamento relevante sobre as melhores práticas atuais para garantir que os conteúdos Web sejam acessíveis, tanto quanto possível, a esta comunidade. Os metadados podem ajudar os utilizadores a encontrar conteúdos mais adequados às suas necessidades.

Documentos de Apoio das WCAG 2.2

O documento WCAG 2.2 foi concebido para responder às necessidades de quem necessita de uma norma técnica estável e passível de referência. Outros documentos, designados documentos de apoio, baseiam-se no documento WCAG 2.2 e abordam outros objetivos importantes, incluindo a possibilidade de serem atualizados para descrever a forma como as WCAG podem ser aplicadas a novas tecnologias. Os documentos de apoio incluem:

  1. Como Cumprir as WCAG 2.2 - Uma referência rápida personalizável das WCAG 2.2 que inclui todas as diretrizes, critérios de sucesso e técnicas para utilização pelos autores durante o desenvolvimento e avaliação de conteúdos Web. Inclui conteúdo das WCAG 2.0, 2.1 e 2.2, podendo ser filtrado de várias formas para ajudar os autores a concentrarem-se no conteúdo relevante.

  2. Compreender as WCAG 2.2 - Um guia para compreender e implementar as WCAG 2.2. Existe um documento curto de “Compreensão” para cada diretriz e critério de sucesso das WCAG 2.2, bem como para tópicos fundamentais.

  3. Técnicas para as WCAG 2.2 - Uma coleção de técnicas e falhas comuns, cada uma apresentada num documento separado que inclui descrição, exemplos, código e testes.

  4. Os Documentos WCAG 2 - Uma breve introdução aos documentos de apoio das WCAG 2 e às orientações suplementares.

  5. Novidades nas WCAG 2.2 apresenta os novos critérios de sucesso através de citações de personas que ilustram os problemas de acessibilidade.

Consulte a Visão Geral das Diretrizes de Acessibilidade para Conteúdo Web (WCAG) para obter uma descrição dos materiais de apoio das WCAG 2.2, incluindo recursos educativos relacionados com as WCAG 2. Recursos adicionais sobre temas como a justificação de negócio para a acessibilidade Web, o planeamento da implementação para melhorar a acessibilidade de sítios Web e políticas de acessibilidade estão disponíveis em Recursos WAI.

Requisitos para as WCAG 2.2

As WCAG 2.2 cumprem um conjunto de requisitos para as WCAG 2.2 que, por sua vez, herdam requisitos das versões anteriores das WCAG 2. Os requisitos estruturam o enquadramento global das diretrizes e asseguram a retrocompatibilidade. O Working Group utilizou também um conjunto menos formal de critérios de aceitação para os critérios de sucesso, com o objetivo de garantir que estes mantêm um estilo e um nível de qualidade semelhantes aos das WCAG 2.0. Estes requisitos limitaram aquilo que podia ser incluído nas WCAG 2.2. Esta limitação foi importante para preservar a natureza das WCAG 2.2 enquanto versão incremental das WCAG 2.

Comparação com as WCAG 2.1

As WCAG 2.2 foram iniciadas com o objetivo de dar continuidade ao trabalho das WCAG 2.1: melhorar as orientações de acessibilidade para três grandes grupos: utilizadores com incapacidades cognitivas ou dificuldades de aprendizagem, utilizadores com baixa visão e utilizadores com incapacidades em dispositivos móveis. Foram propostas e avaliadas muitas formas de responder a estas necessidades, tendo um conjunto dessas propostas sido refinado pelo Working Group. Os requisitos estruturais herdados das WCAG 2.0, a clareza e o impacto das propostas, bem como o calendário disponível, conduziram ao conjunto final de critérios de sucesso incluídos nesta versão. O Working Group considera que as WCAG 2.2 representam um avanço incremental nas orientações de acessibilidade para conteúdo Web em todas estas áreas, mas sublinha que estas diretrizes não satisfazem todas as necessidades dos utilizadores.

As WCAG 2.2 baseiam-se nas WCAG 2.1 e mantêm retrocompatibilidade com estas, o que significa que as páginas Web conformes com as WCAG 2.2 são, no mínimo, tão acessíveis quanto as páginas conformes com as WCAG 2.1. Foram adicionados requisitos que se baseiam nas versões 2.1 e 2.0. As WCAG 2.2 removeram um critério de sucesso, 4.1.1 Análise Sintática. Os autores que estejam obrigados por políticas a conformidade com as WCAG 2.0 ou 2.1 poderão atualizar os conteúdos para as WCAG 2.2, mas poderão ter de continuar a testar e reportar o critério 4.1.1. Os autores que seguem mais do que uma versão das diretrizes devem ter em atenção as seguintes adições.

Novidades nas WCAG 2.2

As WCAG 2.2 alargam as WCAG 2.1 através da adição de novos critérios de sucesso, de definições que lhes dão suporte e de diretrizes para organizar essas adições. Esta abordagem incremental ajuda a tornar claro que os sítios Web conformes com as WCAG 2.2 também estão conformes com as WCAG 2.1. O Accessibility Guidelines Working Group recomenda que os sítios Web adotem as WCAG 2.2 como novo objetivo de conformidade, mesmo quando as obrigações formais mencionem versões anteriores, de modo a proporcionar uma acessibilidade melhorada e antecipar futuras alterações nas políticas.

Os seguintes critérios de sucesso são novos nas WCAG 2.2:

  • 2.4.11 Foco Não Obscurecido (Mínimo) (AA)
  • 2.4.12 Foco Não Obscurecido (Melhorado) (AAA)
  • 2.4.13 Aspeto do Foco (AAA)
  • 2.5.7 Movimentos de Arrastamento (AA)
  • 2.5.8 Tamanho do Alvo (Mínimo) (AA)
  • 3.2.6 Ajuda Consistente (A)
  • 3.3.7 Entrada Redundante (A)
  • 3.3.8 Autenticação Acessível (Mínimo) (AA)
  • 3.3.9 Autenticação Acessível (Melhorada) (AAA)

Os novos critérios de sucesso podem fazer referência a novos termos que também foram adicionados ao glossário e que fazem parte dos requisitos normativos desses critérios de sucesso.

As WCAG 2.2 introduzem igualmente novas secções que detalham aspetos da especificação suscetíveis de ter impacto na privacidade e na segurança.

Numeração nas WCAG 2.2

Para evitar confusão para os implementadores para os quais a retrocompatibilidade com as versões das WCAG 2 é importante, os novos critérios de sucesso das WCAG 2.2 foram acrescentados no final do conjunto de critérios de sucesso da respetiva diretriz. Isto evita a necessidade de alterar a numeração das secções dos critérios de sucesso das WCAG 2, o que aconteceria se fossem inseridos novos critérios de sucesso entre critérios já existentes na diretriz. No entanto, isso significa que os critérios de sucesso de cada diretriz deixaram de estar agrupados por nível de conformidade. A ordem dos critérios de sucesso dentro de cada diretriz não implica qualquer informação sobre o nível de conformidade; apenas o indicador de nível de conformidade (A / AA / AAA) presente no próprio critério de sucesso fornece essa informação. A Referência Rápida das WCAG 2.2 disponibiliza uma forma de visualizar os critérios de sucesso agrupados por nível de conformidade, juntamente com muitas outras opções de filtragem e ordenação.

Conformidade com as WCAG 2.2

As WCAG 2.2 utilizam o mesmo modelo de conformidade das WCAG 2.0. Pretende-se que os sítios Web conformes com as WCAG 2.2 também estejam conformes com as WCAG 2.0 e as WCAG 2.1, o que significa que cumprem os requisitos de quaisquer políticas que façam referência às WCAG 2.0 ou às WCAG 2.1, ao mesmo tempo que respondem melhor às necessidades dos utilizadores da Web atual.

Versões Futuras das Diretrizes de Acessibilidade

Em paralelo com as WCAG 2.2, o Accessibility Guidelines Working Group está a desenvolver outra versão principal das diretrizes de acessibilidade. Espera-se que o resultado deste trabalho constitua uma reestruturação mais substancial das orientações de acessibilidade web do que seria realista para versões incrementais das WCAG 2. O trabalho segue uma metodologia de conceção centrada no utilizador e orientada pela investigação, com o objetivo de produzir o resultado mais eficaz e flexível possível, incluindo os papéis da criação de conteúdos, do suporte dos agentes de utilizador e do suporte das ferramentas de autoria. Trata-se de um esforço plurianual, pelo que as WCAG 2.2 são necessárias como medida intermédia para fornecer orientações de acessibilidade web atualizadas que reflitam as mudanças ocorridas na Web desde a publicação das WCAG 2.0. O Working Group poderá também desenvolver versões intermédias adicionais, dando continuidade às WCAG 2.2, num calendário semelhante e relativamente curto, para fornecer apoio adicional enquanto a versão principal não estiver concluída.

Percetível

A informação e os componentes da interface do utilizador têm de ser apresentados aos utilizadores de formas que estes consigam percecionar.

Alternativas em Texto

Fornecer alternativas em texto para qualquer conteúdo não textual, para que possa ser convertido noutras formas de que as pessoas necessitem, tais como impressão ampliada, braille, fala, símbolos ou linguagem mais simples.

Multimédia Baseada no Tempo

Fornecer alternativas para multimédia baseada no tempo.

Adaptável

Criar conteúdos que possam ser apresentados de diferentes formas (por exemplo, com uma disposição mais simples) sem perda de informação ou de estrutura.

Distinguível

Tornar mais fácil para os utilizadores ver e ouvir os conteúdos, incluindo separar o primeiro plano do fundo.

Operável

Os componentes da interface do utilizador e a navegação têm de ser operáveis.

Acessível por Teclado

Disponibilizar toda a funcionalidade através do teclado.

Tempo Suficiente

Disponibilizar aos utilizadores tempo suficiente para ler e utilizar os conteúdos.

Convulsões e Reações Físicas

Não conceber conteúdos de forma que seja conhecida por provocar convulsões ou reações físicas.

Navegável

Disponibilizar formas de ajudar os utilizadores a navegar, encontrar conteúdos e determinar onde se encontram.

Modalidades de Entrada

Tornar mais fácil para os utilizadores operar funcionalidades através de vários modos de entrada para além do teclado.

Compreensível

A informação e o funcionamento da interface do utilizador têm de ser compreensíveis.

Legível

Tornar o conteúdo textual legível e compreensível.

Previsível

Fazer com que as páginas Web apareçam e funcionem de formas previsíveis.

Assistência ao Preenchimento

Ajudar os utilizadores a evitar e corrigir erros.

Robusto

O conteúdo tem de ser suficientemente robusto para poder ser interpretado por uma grande variedade de agentes de utilizador, incluindo tecnologias de apoio.

Compatível

Maximizar a compatibilidade com agentes de utilizador atuais e futuros, incluindo tecnologias de apoio.

Conformidade

Esta secção apresenta os requisitos de conformidade com as WCAG 2.2. Também fornece informações sobre como fazer declarações de conformidade, as quais são opcionais. Por fim, descreve o significado de ser suportado em termos de acessibilidade, uma vez que apenas formas de utilização de tecnologias suportadas em termos de acessibilidade podem ser consideradas fiáveis para efeitos de conformidade. Compreender a Conformidade inclui explicações adicionais sobre o conceito de suporte em termos de acessibilidade.

Interpretação dos Requisitos Normativos

O conteúdo principal das WCAG 2.2 é normativo e define requisitos que têm impacto nas declarações de conformidade. O material introdutório, os anexos, as secções assinaladas como “não normativas”, os diagramas, os exemplos e as notas são informativos (não normativos). O material não normativo fornece informação consultiva para ajudar na interpretação das diretrizes, mas não cria requisitos que tenham impacto numa declaração de conformidade.

As palavras-chave MAY, MUST, MUST NOT, NOT RECOMMENDED, RECOMMENDED, SHOULD e SHOULD NOT devem ser interpretadas conforme descrito em [[RFC2119]].

Requisitos de Conformidade

Para que uma página Web esteja conforme com as WCAG 2.2, todos os seguintes requisitos de conformidade têm de ser satisfeitos:

Nível de Conformidade

Um dos seguintes níveis de conformidade é integralmente satisfeito.

  • Para conformidade de Nível A (o nível mínimo de conformidade), a página Web satisfaz todos os critérios de sucesso de Nível A, ou é fornecida uma versão alternativa conforme.
  • Para conformidade de Nível AA, a página Web satisfaz todos os critérios de sucesso de Nível A e de Nível AA, ou é fornecida uma versão alternativa conforme de Nível AA.
  • Para conformidade de Nível AAA, a página Web satisfaz todos os critérios de sucesso de Nível A, Nível AA e Nível AAA, ou é fornecida uma versão alternativa conforme de Nível AAA.

Embora a conformidade só possa ser alcançada nos níveis indicados, os autores são incentivados a reportar (na sua declaração) qualquer progresso no cumprimento de critérios de sucesso de níveis superiores ao nível de conformidade alcançado.

Não é recomendado exigir conformidade de Nível AAA como política geral para sítios Web completos, porque não é possível satisfazer todos os critérios de sucesso de Nível AAA para alguns conteúdos.

Páginas completas

A conformidade (e o nível de conformidade) aplica-se apenas a página(s) Web completas e não pode ser alcançada se parte de uma página Web for excluída.

Para efeitos de determinação da conformidade, as alternativas a parte do conteúdo de uma página são consideradas parte da página quando podem ser obtidas diretamente a partir da mesma, por exemplo, uma descrição longa ou uma apresentação alternativa de um vídeo.

Os autores de páginas Web que não possam estar conformes devido a conteúdos fora do seu controlo podem considerar uma Declaração de Conformidade Parcial.

Uma página completa inclui cada variação da página que seja automaticamente apresentada para diferentes tamanhos de ecrã (por exemplo, variações numa página Web responsiva). Cada uma dessas variações tem de estar conforme (ou ter uma versão alternativa conforme) para que a página inteira esteja conforme.

Processos completos

Quando uma página Web é uma de uma série de páginas Web que apresentam um processo (isto é, uma sequência de passos que têm de ser concluídos para realizar uma atividade), todas as páginas Web do processo têm de estar conformes no nível especificado ou superior. (A conformidade não é possível num determinado nível se alguma página do processo não estiver conforme nesse nível ou superior.)

Uma loja online possui uma série de páginas utilizadas para selecionar e comprar produtos. Todas as páginas da série, do início ao fim (checkout), têm de estar conformes para que qualquer página que faça parte do processo esteja conforme.

Apenas Formas de Utilização de Tecnologias Suportadas em termos de Acessibilidade

Apenas formas de utilização de tecnologias suportadas em termos de acessibilidade podem ser consideradas fiáveis para satisfazer os critérios de sucesso. Qualquer informação ou funcionalidade disponibilizada de uma forma que não seja suportada em termos de acessibilidade também tem de estar disponível de uma forma que seja suportada em termos de acessibilidade. (Consulte Compreender o suporte em termos de acessibilidade.)

Não Interferência

Se tecnologias forem utilizadas de uma forma que não seja suportada em termos de acessibilidade, ou se forem utilizadas de uma forma não conforme, então não podem bloquear a capacidade dos utilizadores acederem ao restante conteúdo da página. Além disso, a página Web como um todo continua a satisfazer os requisitos de conformidade em cada uma das seguintes condições:

  1. quando qualquer tecnologia que não seja considerada fiável estiver ativada num agente de utilizador,
  2. quando qualquer tecnologia que não seja considerada fiável estiver desativada num agente de utilizador, e
  3. quando qualquer tecnologia que não seja considerada fiável não for suportada por um agente de utilizador

Além disso, os seguintes critérios de sucesso aplicam-se a todo o conteúdo da página, incluindo conteúdo que não seja de outra forma considerado fiável para satisfazer a conformidade, porque a falha no seu cumprimento pode interferir com qualquer utilização da página:

  • 1.4.2 - Controlo de Áudio,
  • 2.1.2 - Sem Bloqueio de Teclado,
  • 2.3.1 - Três Intermitências ou Abaixo do Limite, e
  • 2.2.2 - Pausar, Parar, Ocultar.

Se uma página não puder estar conforme (por exemplo, uma página de teste de conformidade ou uma página de exemplo), não pode ser incluída no âmbito da conformidade nem numa declaração de conformidade.

Para mais informações, incluindo exemplos, consulte Compreender os Requisitos de Conformidade.

Declarações de Conformidade (Opcional)

A conformidade é definida apenas para páginas Web. No entanto, pode ser feita uma declaração de conformidade para abranger uma página, uma série de páginas ou múltiplas páginas Web relacionadas.

Componentes Obrigatórios de uma Declaração de Conformidade

As declarações de conformidade não são obrigatórias. Os autores podem estar conformes com as WCAG 2.2 sem fazer uma declaração. Contudo, se for feita uma declaração de conformidade, esta tem de incluir as seguintes informações:

  1. Data da declaração
  2. Título, versão e URI das diretrizes “Diretrizes de Acessibilidade para Conteúdo Web 2.2 em https://www.w3.org/TR/WCAG22/”
  3. Nível de conformidade satisfeito: (Nível A, AA ou AAA)

  4. Uma descrição concisa das páginas Web, tal como uma lista de URIs às quais a declaração se aplica, incluindo se os subdomínios estão incluídos na declaração.

    As páginas Web podem ser descritas através de uma lista ou por uma expressão que descreva todos os URIs incluídos na declaração.

    Produtos baseados na Web que não tenham um URI antes da instalação no sítio Web do cliente podem incluir uma declaração de que o produto estaria conforme após instalação.

  5. Uma lista das tecnologias de conteúdo Web consideradas fiáveis.

Se for utilizado um logótipo de conformidade, este constitui uma declaração e tem de ser acompanhado pelos componentes obrigatórios de uma declaração de conformidade listados acima.

Componentes Opcionais de uma Declaração de Conformidade

Para além dos componentes obrigatórios de uma declaração de conformidade acima indicados, considere fornecer informações adicionais para ajudar os utilizadores. As informações adicionais recomendadas incluem:

  • Uma lista de critérios de sucesso, para além do nível de conformidade declarado, que tenham sido satisfeitos. Esta informação deve ser fornecida numa forma utilizável pelos utilizadores, preferencialmente através de metadados legíveis por máquina.
  • Uma lista das tecnologias específicas que são “ utilizadas mas não consideradas fiáveis”.
  • Uma lista dos agentes de utilizador, incluindo tecnologias de apoio, que foram utilizados para testar o conteúdo.
  • Uma lista das características específicas de acessibilidade do conteúdo, fornecida através de metadados legíveis por máquina.
  • Informação sobre quaisquer medidas adicionais tomadas para além dos critérios de sucesso com o objetivo de melhorar a acessibilidade.
  • Uma versão em metadados legíveis por máquina da lista de tecnologias específicas que são consideradas fiáveis.
  • Uma versão em metadados legíveis por máquina da declaração de conformidade.

Consulte Compreender as Declarações de Conformidade para mais informações e exemplos de declarações de conformidade.

Consulte Compreender os Metadados para mais informações sobre a utilização de metadados em declarações de conformidade.

Declaração de Conformidade Parcial - Conteúdo de Terceiros

As páginas Web às quais venha posteriormente a ser adicionado conteúdo adicional podem utilizar uma “declaração de conformidade parcial”. Por exemplo, um programa de correio eletrónico, um blogue, um artigo que permita aos utilizadores adicionar comentários ou aplicações que suportem conteúdo contribuído pelos utilizadores. Outro exemplo seria uma página, como um portal ou um sítio de notícias, composta por conteúdo agregado de múltiplos contribuidores, ou sítios que inserem automaticamente conteúdo proveniente de outras fontes ao longo do tempo, como acontece quando anúncios são inseridos dinamicamente.

Nestes casos, não é possível saber, no momento da publicação original, qual será o conteúdo não controlado das páginas. É importante notar que o conteúdo não controlado também pode afetar a acessibilidade do conteúdo controlado. Existem duas opções disponíveis:

  1. Pode ser feita uma determinação de conformidade com base no melhor conhecimento disponível. Se uma página deste tipo for monitorizada e corrigida (o conteúdo não conforme for removido ou tornado conforme) no prazo de dois dias úteis, então pode ser feita uma determinação ou declaração de conformidade, uma vez que, exceto erros em conteúdos contribuídos externamente que sejam corrigidos ou removidos quando detetados, a página está conforme. Não pode ser feita qualquer declaração de conformidade se não for possível monitorizar ou corrigir conteúdos não conformes;

    OU

  2. Pode ser feita uma “declaração de conformidade parcial” indicando que a página não está conforme, mas que poderia estar conforme se determinadas partes fossem removidas. A forma dessa declaração seria: “Esta página não está conforme, mas estaria conforme com as WCAG 2.2 no nível X se as seguintes partes provenientes de fontes não controladas fossem removidas.” Além disso, o seguinte também teria de ser verdadeiro relativamente ao conteúdo não controlado descrito na declaração de conformidade parcial:

    1. Não se trata de conteúdo sob controlo do autor.
    2. É descrito de uma forma que os utilizadores conseguem identificar (por exemplo, não pode ser descrito como “todas as partes que não controlamos”, a menos que estejam claramente identificadas como tal.)

Declaração de Conformidade Parcial - Idioma

Pode ser feita uma “declaração de conformidade parcial devido ao idioma” quando a página não está conforme, mas estaria conforme se existisse suporte em termos de acessibilidade para (todos) os idiomas utilizados na página. A forma dessa declaração seria: “Esta página não está conforme, mas estaria conforme com as WCAG 2.2 no nível X se existisse suporte em termos de acessibilidade para o(s) seguinte(s) idioma(s):”
